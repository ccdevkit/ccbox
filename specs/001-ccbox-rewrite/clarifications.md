# Clarifications Log: ccbox — Docker-Sandboxed Claude Code Runner

Decision log from spec clarification sessions. Answers here are encoded into the feature spec as functional requirements; this file preserves the rationale and traceability.

## Session 2026-03-25

- Q: Is the passthrough proxy the only execution channel from container to host, or can the container reach the host through other mechanisms? → A: The passthrough proxy is the only container→host execution channel; only explicitly configured command prefixes are routed. The container has no Docker socket mount, SSH back-channel, or other mechanism to execute commands on the host. *(Encoded in FR-030)*
- Q: Should passthrough commands use the original launch directory or the container's current working directory at invocation time? → A: The container process's current working directory at command invocation time (dynamic). Mechanism deferred to planning. *(Encoded in FR-008a)*
- Q: How should ccbox handle Unix signals (SIGINT, SIGTERM, SIGHUP) received by the host process? → A: Forward signals to the container process, wait for the container to exit, then propagate its exit code. *(Encoded in FR-035)*
- Q: What should ccbox do when the passthrough proxy encounters a host-side command failure (e.g., command not found)? → A: Return the error output and non-zero exit code to the container process silently, identical to a local command failure. *(Encoded in FR-036)*
- Q: Should ccbox auto-clean old Docker images, and should it offer a cleanup subcommand? → A: Images created via `--use` (pinned) are retained; images created via auto-update are auto-removed when a newer auto-update image is built. `ccbox clean` removes all ccbox images except the latest. *(Encoded in FR-037, FR-038)*
- Q: When the container's network connection drops mid-session, should ccbox attempt recovery? → A: No — ccbox does not intervene. Network resilience is Claude Code's responsibility. *(Encoded in Edge Cases)*
- Q: When the user passes `--append-system-prompt`, should ccbox skip its own system prompt injection or always inject additively? → A: Always additive — ccbox injects its system prompt alongside any user-provided prompt. The product spec §5.9 statement about skipping injection is incorrect; FR-024 is the canonical behavior. *(Encoded in FR-024)*
- Q: The product spec (§11 limitation #4) says "No old image cleanup" while the feature spec (FR-037/FR-038) describes auto-cleanup and `ccbox clean`. Which is correct? → A: Feature spec is correct — auto-cleanup and `ccbox clean` are intended features of this rewrite. The product spec limitation describes old behavior being replaced. *(Encoded in FR-037, FR-038)*
- Q: Should the feature spec include a functional requirement for statusline proxying (described in product spec §5.8 and technical spec §4.4)? → A: No — statusline proxying is covered by the passthrough feature. It is not a separate feature and all references should be removed from the feature spec. *(Out of scope — covered by passthrough)*
- Q: Should the passthrough `[NOTE: This command was run on the host machine]` annotation appear on both success and failure, or only on success? → A: Always — the note appears on all passthrough output (success and failure). FR-036's "no additional annotation" means no extra error-specific handling beyond the standard note. *(Encoded in FR-036)*
- Q: When `--` is absent, does ccbox parse its own flags or forward everything to Claude Code? → A: Without `--`, all arguments are parsed as ccbox flags. The `--` separator is required to pass any arguments to Claude Code. Examples: `ccbox -pt:git` → ccbox gets `-pt:git`, Claude gets no args; `ccbox -- -p "Hello"` → Claude gets `-p "Hello"`; `ccbox -pt:git -- -p "Hello"` → ccbox gets `-pt:git`, Claude gets `-p "Hello"`. *(Encoded in FR-006)*
- Q: Should FR-022 detect paths heuristically (by pattern/extension) or semantically (by argument type)? → A: Semantically. ccbox parses Claude CLI arguments to determine which expect a file path, then makes existing files available to the container. No heuristic pattern-matching on extensions or path separators. *(Encoded in FR-022)*
- Q: Is FR-023 (`--append-system-prompt` / `--append-system-prompt-file` support) a separate feature or redundant? → A: Redundant. ccbox does not have its own `--append-system-prompt*` flags. File mounting is handled by FR-022 (semantic arg parsing); additive prompt injection is handled by FR-024. FR-023 removed. *(FR-023 removed from spec)*
- Q: Does FR-011's image path detection mechanism need to be specified in the spec? → A: No — the *what* is sufficient; mechanism deferred to planning. *(No spec change)*
- Q: Is FR-003's credential extraction redundant with FR-028's `~/.claude/` mount? → A: No — `~/.claude/` does not reliably contain auth state. FR-003's active extraction is a separate, necessary requirement. *(No spec change — FR-003 retained)*
- Q: Should the "Max subscription users may receive Sonnet" assumption remain in the spec? → A: No — removed. This is a Claude Code upstream behavior, not a ccbox concern. *(Removed from spec)*
